# Module 5: Design Patterns & Best Practices

## 1. Writing good tool descriptions

Since the model chooses which tool to call based only on the description text — not your source code — this really is prompt engineering, and it's the highest-leverage thing you can do. Anthropic's own guidance treats it as the most important factor in tool performance: your descriptions should explain every detail about the tool, including what it does, when to use it, what parameters mean, and what it returns — aim for at least 3–4 sentences per tool, more if the tool is complex.

Concretely, a good description has three parts:
- **A specific action statement** — "Retrieves the 10 most recent commits from a GitHub repository branch," not "gets data."
- **Parameter constraints** — required vs. optional fields, valid ranges, formats.
- **Output shape** — what the model should expect back, so it can decide whether the result actually answers the request.

A few sharp edges worth knowing:
- **Front-load the important part.** Agents may not read a whole multi-sentence description, so lead with what the tool does, not caveats or setup steps.
- **Error messages are part of the description's job too.** "Repository not found" gives the model nothing to act on. "Repository 'owner/repo' not found — verify the name or check if it's private" gives it a recovery path.
- **This is measurable, not just a feeling.** Precise refinements to tool descriptions alone produced dramatic, measurable improvements on Claude's tool-use benchmarks — treat tool descriptions the way you'd treat prompts you iterate on and eval against real usage.

## 2. Scoping tools narrowly vs. broadly

There's a real tradeoff here, not a default "more tools is better":

| Too narrow | Too broad |
|---|---|
| Dozens of near-identical tools (`get_user_by_id`, `get_user_by_email`, `get_user_by_username`) | One mega-tool with 40 optional parameters covering every case |
| Model has to guess which near-duplicate fits | Model has to guess which parameters matter for this call |
| Selection accuracy drops as tool count grows and options overlap | Schema becomes unreadable; errors get harder to diagnose |

The practical fix is **consolidation around distinct actions, not around resources** — one `search_users` tool with a good `filter_by` parameter beats five thin wrapper tools. And when two tools *do* need to coexist, their descriptions must clearly differentiate purpose and boundaries, or the model will misroute between them (a documented failure mode: tools like `analyze_content` vs. `analyze_document` with near-identical descriptions cause exactly this).

As a rule of thumb: every additional tool costs a slice of the model's attention at selection time, permanently, on every turn it's loaded — so each one needs to earn its place.

## 3. Read-only vs. mutating tools

MCP has a standard vocabulary for this: **tool annotations**, four boolean hints on each tool definition.

| Annotation | Meaning | Default |
|---|---|---|
| `readOnlyHint` | Tool doesn't modify its environment | `false` |
| `destructiveHint` | May perform irreversible/destructive updates (only meaningful if not read-only) | `true` |
| `idempotentHint` | Calling it twice with the same args has no extra effect — safe to retry | `false` |
| `openWorldHint` | Talks to something outside the server's own closed system (the internet, a third-party API) | `true` |

The defaults are deliberately conservative — an unannotated tool is treated as destructive and external, which is the safer failure mode. The practical payoff: hosts can auto-approve read-only tools and require explicit human confirmation before running anything marked destructive.

Two things worth being precise about:
- **These are hints, not guarantees.** A malicious or buggy server could mark a destructive tool `readOnlyHint: true` to skip confirmation — so annotations should drive UX (which dialog to show), never be the actual security boundary.
- **A tool can't honestly be both.** If `readOnlyHint` is true, `destructiveHint`/`idempotentHint` don't apply — don't set contradictory hints.

## 4. Handling large outputs

Nothing in the base protocol pages tool results automatically — an unbounded `query_db` or `list_files` tool will happily return 50,000 rows straight into the model's context window if you let it. Standard mitigations:

- **Paginate at the tool level.** Accept a `cursor`/`page` and `limit` parameter, return a `next_cursor` in the result, and say so explicitly in the description — the model needs to know pagination exists to use it correctly, since operational details like pagination limits are exactly the kind of thing agents rely on the description to communicate.
- **Summarize instead of dumping.** For tools that wrap large documents or datasets, consider returning a summary plus a way to drill in (a second tool call, or a resource URI) rather than the whole payload.
- **Cap and truncate defensively**, even if the caller doesn't ask for a limit — a tool that can return unbounded output is a tool that can blow the context window on a single call, degrading everything else in the conversation.
- **Prefer resources for large reference data.** If something is meant to be read rather than acted on (a schema, a doc, a config file), exposing it as a `resource` rather than a `tool` result keeps it out of the model's working context until it's actually attached.

## 5. Security

- **Least-privilege access.** A server should only be able to do what its declared tools claim to do — scope database credentials to read-only where possible, scope filesystem access to specific directories (not full home-directory access), scope API tokens to the narrowest permission set the tool needs.
- **Sandboxing.** Especially for stdio servers, remember the server runs as a subprocess with your full user privileges — anything that process can read or write, the model can effectively read or write too. Treat local MCP servers with the same suspicion you'd treat any third-party code you're about to execute locally.
- **Validate inputs from the model before executing.** The model constructs tool arguments from a schema, but a schema is not a guarantee — always validate on the server side (type-check, bounds-check, sanitize anything going into a shell command or SQL query) rather than trusting that "it matched the JSON Schema" means it's safe to execute. This is the same discipline you'd apply to any untrusted user input, because functionally, that's what model-generated arguments are.
- **Treat annotations as UX signals, not access control**, per above — don't grant broader capabilities to a tool just because it claims `openWorldHint: false`.

## Examples 
### 6. Tool descriptions — bad vs. good

**Bad:**
```python
@mcp.tool()
def get_data(id: str):
    """Gets data."""
```

**Good:**
```python
@mcp.tool()
def get_campaign_performance(campaign_id: str, date_range: str = "last_7_days"):
    """
    Retrieve performance metrics for a single ad campaign.

    Use this when the user asks about how a specific campaign is performing
    (impressions, clicks, spend, conversions) — not for comparing multiple
    campaigns, use `compare_campaigns` for that instead.

    Args:
        campaign_id: The campaign's unique ID, e.g. "camp_8f2a91".
                     Get this from `list_campaigns` if you don't have it.
        date_range: One of "last_7_days", "last_30_days", "this_month".
                     Defaults to "last_7_days".

    Returns:
        JSON with impressions, clicks, spend_usd, conversions, and ctr,
        aggregated over the requested date range. Returns null values
        (not zeros) for metrics unavailable in that range.
    """
```
The second version tells the model *when* to use it, *how it differs* from a similar tool, what a valid ID looks like, and how to interpret a null vs. a zero — all things the model would otherwise have to guess.

**Bad error message:**
```json
{"content": [{"type": "text", "text": "Error: not found"}], "isError": true}
```
**Good error message:**
```json
{"content": [{"type": "text", "text": "Campaign 'camp_8f2a91' not found. It may have been archived — check `list_campaigns` for active IDs, or `list_archived_campaigns` if this one predates last month."}], "isError": true}
```

### 2. Narrow vs. broad scoping

**Too narrow** (five tools doing one job):
```
get_user_by_id(id)
get_user_by_email(email)
get_user_by_username(username)
get_user_by_phone(phone)
get_user_by_external_id(external_id)
```
**Consolidated:**
```python
@mcp.tool()
def get_user(identifier: str, id_type: Literal["id","email","username","phone","external_id"] = "id"):
    """Look up a single user by any known identifier type."""
```

**Too broad** (one tool, unreadable schema):
```python
@mcp.tool()
def manage_user(action: str, id: str = None, email: str = None, name: str = None,
                 role: str = None, delete: bool = False, notify: bool = False,
                 merge_with: str = None, ...):
    """Does things to users."""
```
This forces the model to guess which of 15 optional fields matter for `action="promote"` vs `action="delete"`. Split it:
```python
def update_user_role(user_id: str, new_role: str): ...
def delete_user(user_id: str, notify: bool = True): ...
def merge_users(primary_id: str, duplicate_id: str): ...
```
Each has a tight, self-explaining schema instead of one conditional mess.

### 3. Read-only vs. mutating — annotations in practice

```python
from mcp.types import ToolAnnotations

@mcp.tool(annotations=ToolAnnotations(
    readOnlyHint=True,
    openWorldHint=False,   # internal DB, not an external service
))
def search_orders(customer_id: str) -> list[dict]:
    """Look up orders for a customer. Read-only — safe to call freely."""
    ...

@mcp.tool(annotations=ToolAnnotations(
    readOnlyHint=False,
    destructiveHint=True,   # irreversible
    idempotentHint=True,    # calling twice with same args = same end state
    openWorldHint=True,     # hits a real payment processor
))
def refund_order(order_id: str, amount_usd: float) -> dict:
    """Issue a refund. Irreversible — host should confirm with the user first."""
    ...

@mcp.tool(annotations=ToolAnnotations(
    readOnlyHint=False,
    destructiveHint=False,  # additive, not destructive
    idempotentHint=False,   # calling twice adds two comments
))
def add_order_note(order_id: str, note: str) -> dict:
    """Append an internal note to an order."""
    ...
```
A host like Claude Desktop can then auto-run `search_orders` silently, but pop a confirmation dialog before `refund_order` — driven entirely by these four booleans, no hardcoded tool-name logic required.

### 4. Handling large outputs

**Unbounded (bad):**
```python
@mcp.tool()
def list_transactions(account_id: str) -> list[dict]:
    """Returns every transaction for an account."""
    return db.query(f"SELECT * FROM transactions WHERE account_id = '{account_id}'")
    # could be 200,000 rows → blows the context window on one call
```

**Paginated:**
```python
@mcp.tool()
def list_transactions(account_id: str, cursor: str | None = None, limit: int = 50) -> dict:
    """
    Returns up to `limit` transactions (max 200), newest first.
    Pass the returned `next_cursor` back in to get the next page;
    `next_cursor: null` means there are no more results.
    """
    rows, next_cursor = db.query_page(account_id, cursor, min(limit, 200))
    return {"transactions": rows, "next_cursor": next_cursor}
```

**Summarize instead of dumping** (for a large document/report tool):
```python
@mcp.tool()
def get_report_summary(report_id: str) -> dict:
    """Returns a short summary + section list. Use `get_report_section`
    to fetch one section in full instead of the whole report."""
    return {"summary": summarize(report_id), "sections": list_section_titles(report_id)}
```

**As a resource instead of a tool result**, when it's reference material rather than an action:
```python
@mcp.resource("reports://{report_id}/full")
def read_full_report(report_id: str) -> str:
    """Full report text — attached explicitly by the user, not auto-loaded by the model."""
    return load_report(report_id)
```

### 5. Security

**Least privilege — scope the credential, not just the code:**
```python
# Bad: server has a full-access DB connection string
DB_URL = "postgres://admin:pw@host/db"   # can DROP TABLE

# Good: a dedicated read-only role for this server
DB_URL = "postgres://mcp_readonly:pw@host/db"  # SELECT only, enforced by the DB itself
```

**Sandboxing a local/stdio server** — e.g. restricting filesystem access instead of trusting the process:
```python
ALLOWED_ROOT = Path("/home/user/projects/reports").resolve()

@mcp.tool()
def read_file(path: str) -> str:
    """Read a file. Restricted to the reports/ directory."""
    target = (ALLOWED_ROOT / path).resolve()
    if not target.is_relative_to(ALLOWED_ROOT):
        raise ValueError("Access denied: path is outside the allowed directory")
    return target.read_text()
```

**Validating model-supplied input before it reaches anything dangerous** — schema validation alone isn't enough:
```python
@mcp.tool()
def run_report_query(sql_filter: str) -> list[dict]:
    """Run a filtered report query. `sql_filter` is a WHERE clause fragment."""
    # BAD: string-concatenating model output straight into SQL
    # db.execute(f"SELECT * FROM orders WHERE {sql_filter}")

    # GOOD: never let model output become executable code —
    # use parameterized structure instead of a raw SQL fragment
    if not re.fullmatch(r"[a-zA-Z0-9_]+ (=|>|<) '[^';]+'", sql_filter):
        raise ValueError("Filter must match 'column op value' — no other SQL allowed")
    return db.execute("SELECT * FROM orders WHERE " + sql_filter)
```
Even though `sql_filter` came from a schema-valid tool call, it's still untrusted text — the same discipline as sanitizing any user-submitted string, because that's effectively what it is once the model constructs it.
