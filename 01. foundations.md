# 📘 Model Context Protocol (MCP) — Learning Syllabus

## Module 1: Foundations — Why MCP Exists
- The core problem: every LLM app historically needed **custom, one-off integrations** with each tool/data source (N models × M tools = N×M integrations)
- MCP's pitch: a **standardized protocol** so any MCP-compatible model can talk to any MCP-compatible tool/server — reducing N×M to N+M
- Comparison to prior approaches: function calling / tool use APIs vs. MCP as a *protocol layer* on top of that concept
- Client-server architecture: **Host** (the app, e.g. Claude Desktop) → **Client** (maintains 1:1 connection) → **Server** (exposes tools/data)

## Module 2: Core Architecture
- **Transport layer**: stdio (local processes) vs. HTTP/SSE (remote servers)
- **JSON-RPC 2.0** as the underlying message format
- The three primitives servers can expose:
  - **Tools** — functions the model can call (actions)
  - **Resources** — data the model can read (files, DB records, API responses)
  - **Prompts** — reusable prompt templates the server provides
- Capability negotiation — how client/server agree on what's supported at connection time

## Module 3: Building an MCP Server
- Choosing an SDK (Python, TypeScript, and others in the official SDK list)
- Defining a tool: name, description, input schema (JSON Schema), handler function
- Defining a resource: URI scheme, content type, read handler
- Local dev loop: running a server over stdio, testing with the MCP Inspector tool
- Error handling and returning structured results back to the model

## Module 4: Connecting a Server to a Client
- Configuring a host app (e.g. Claude Desktop, Claude Code) to launch/connect to your server
- Local servers (subprocess-based) vs. remote/hosted servers (auth, OAuth flows)
- Debugging connection issues — capability mismatches, schema validation errors

## Module 5: Design Patterns & Best Practices
- Writing good **tool descriptions** — this is effectively prompt engineering, since the model chooses tools based on the description text alone
- Scoping tools narrowly vs. broadly — too many tools can hurt selection accuracy
- Read-only vs. mutating tools — when to require confirmation before an action executes
- Handling large outputs (pagination, summarization) so tool results don't blow the context window
- Security: least-privilege access, sandboxing, validating inputs from the model before executing

## Module 6: Real-World Ecosystem
- Reference servers (filesystem, git, databases, Slack, etc.) as implementation examples
- The MCP registry / directory concept — how discovery works
- How this differs from and complements Anthropic's other "connector" concepts you've seen referenced (e.g. Google Drive, Slack integrations)

## Module 7: Advanced Topics
- Multi-server orchestration — a host connecting to several servers simultaneously
- Sampling — servers requesting a completion *from* the client's LLM (inverting the usual direction)
- Streaming responses over SSE
- Versioning and backward compatibility as the spec evolves
