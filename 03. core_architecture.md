# Module 2: Core Architecture

## Overview

Module 1 established *why* MCP exists and the high-level Host → Client → Server shape. This module goes one level deeper: **how bytes actually move between a client and a server**, what message format they speak, what a server is allowed to expose, and how the two sides figure out what each supports before any real work happens.

By the end of this module you should be able to answer: *if I start a client and a server, what actually happens on the wire before the model ever calls a tool?*


## The Transport Layer

The transport is the literal pipe the client and server talk through. MCP defines two standard transports — the choice depends entirely on **where the server lives relative to the host**.

### stdio (Standard Input/Output) — for local processes

The client spawns the server as a **local subprocess** and communicates over its standard input and output streams.

```
┌──────────┐   spawns    ┌──────────────┐
│  Client  │────────────▶│   Server     │
│ (in Host)│              │ (subprocess) │
└──────────┘              └──────────────┘
      │      stdin/stdout       │
      └────────◀──────────────▶┘
       JSON-RPC messages, one per line
```

- The server is a program on the **same machine** — a script, a binary, anything the client can launch as a child process
- Messages are exchanged as newline-delimited JSON over stdin/stdout
- **No network involved** — no ports, no auth handshake, no latency beyond process I/O
- Lifecycle is tied to the client: when the client disconnects, it typically terminates the subprocess

**When to use it:** local tools — a filesystem server, a local git server, a server wrapping a CLI tool already installed on the machine. This is the simplest possible transport and the one most examples/tutorials start with.

Here's a concrete, worked example — a Python client spawning a tiny MCP server as a subprocess, with the actual bytes that cross stdin/stdout.

#### Example: A Client Spawning a Local Server

**The server** — a minimal script exposing one tool (`add`), reading JSON-RPC requests line-by-line from stdin and writing responses to stdout:

```python
# server.py
import sys, json

def handle(msg):
    if msg["method"] == "initialize":
        return {"jsonrpc": "2.0", "id": msg["id"],
                "result": {"capabilities": {"tools": {}}}}
    if msg["method"] == "tools/call" and msg["params"]["name"] == "add":
        a, b = msg["params"]["arguments"]["a"], msg["params"]["arguments"]["b"]
        return {"jsonrpc": "2.0", "id": msg["id"],
                "result": {"content": [{"type": "text", "text": str(a + b)}]}}

for line in sys.stdin:
    request = json.loads(line)
    response = handle(request)
    if response:
        print(json.dumps(response), flush=True)   # one JSON object per line
```

**The client** — spawns `server.py` as a subprocess and talks to it purely through its stdin/stdout pipes:

```python
# client.py
import subprocess, json

proc = subprocess.Popen(
    ["python3", "server.py"],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    text=True
)

def send(msg):
    proc.stdin.write(json.dumps(msg) + "\n")
    proc.stdin.flush()
    return json.loads(proc.stdout.readline())

# 1. Handshake
print(send({"jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {}}))

# 2. Call the tool
print(send({
    "jsonrpc": "2.0", "id": 2, "method": "tools/call",
    "params": {"name": "add", "arguments": {"a": 4, "b": 7}}
}))
```

#### What Actually Crosses the Pipe

If you logged every line sent and received, it would look like this — four newline-delimited JSON objects total, no HTTP, no ports, no network stack involved:

```
CLIENT → SERVER (stdin):
{"jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {}}

SERVER → CLIENT (stdout):
{"jsonrpc": "2.0", "id": 1, "result": {"capabilities": {"tools": {}}}}

CLIENT → SERVER (stdin):
{"jsonrpc": "2.0", "id": 2, "method": "tools/call", "params": {"name": "add", "arguments": {"a": 4, "b": 7}}}

SERVER → CLIENT (stdout):
{"jsonrpc": "2.0", "id": 2, "result": {"content": [{"type": "text", "text": "11"}]}}
```

#### Why This Example Makes the Bullet Points Concrete

- **"Spawns as a subprocess"** — `subprocess.Popen([...])` is literally that: the client's process tree now has the server as a child process. If you ran `ps` while this executed, you'd see `python3 server.py` nested under the client.
- **"Newline-delimited JSON"** — notice `json.dumps(msg) + "\n"` on the client and `for line in sys.stdin` on the server. Each message is exactly one line; that's the entire framing protocol — no length prefixes, no delimiters beyond `\n`.
- **"No network involved"** — there's no `socket`, no `host`/`port` anywhere in this code. Data moves through OS-level pipes connecting the two processes' file descriptors.
- **"Lifecycle tied to the client"** — if `client.py` exits or crashes, `proc` (the server subprocess) typically gets torn down with it, since it's a child process rather than an independent long-running service. Real MCP clients call `proc.terminate()` explicitly on disconnect for a clean shutdown.

This is intentionally simplified — a real MCP server would use the official SDK (which handles the JSON-RPC plumbing, schema validation, and the full `initialize` capability payload for you) rather than hand-rolling `sys.stdin` parsing like this. But the underlying mechanics — spawn, write a line, read a line — are exactly what's happening underneath the SDK too.

### HTTP with SSE (Server-Sent Events) — for remote servers

The client connects to a server running **elsewhere** — a different machine, a hosted service — over standard HTTP, with **Server-Sent Events** used to stream messages from server to client.

```
┌──────────┐    HTTP POST (client → server requests)   ┌──────────┐
│  Client  │───────────────────────────────────────────▶│  Server  │
│ (in Host)│                                             │ (remote) │
│          │◀───────────────────────────────────────────│          │
└──────────┘   SSE stream (server → client responses,    └──────────┘
                notifications, streaming results)
```

- The server is a **long-running network service**, reachable over a URL, potentially serving many different clients at once
- **Client → server**: regular HTTP POST requests carry outgoing messages
- **Server → client**: an SSE stream lets the server push messages back — responses, notifications, or streamed partial results — without the client having to poll
- Because it's a network connection, this transport is where **authentication** (API keys, OAuth) becomes relevant — stdio has no such concept since it never leaves the local machine

**Why SSE ?** The simple answer: When the client and server are on the same computer (stdio), they can just pass messages back and forth directly, like handing notes to someone sitting next to you. But when the server lives somewhere else — a different machine, a hosted service — that direct hand-off isn't possible anymore, and a normal web request only gives you one reply before the connection closes, which isn't enough if the server needs to send back several things over time, or wants to speak up on its own without being asked first. So MCP splits the conversation into two one-way lanes: the client sends its requests over regular POST messages, dropping them off without waiting around, while a separate connection — the SSE stream — stays open the whole time so the server can push back whatever it needs, whenever it's ready, without the client having to keep asking "got anything yet?" It's basically the difference between constantly checking your phone for a text versus just leaving the chat open and letting replies pop in on their own.

**When to use it:** hosted/shared servers — a company-wide Slack or database server that many users' clients connect to, or any server you don't want to (or can't) run as a local subprocess.

Here's the same worked-example treatment, but over the network — a Flask server exposing an SSE endpoint plus a POST endpoint, and a client authenticating with an API key.

#### Example: A Client Connecting to a Remote Server

**The server** — a long-running service with two endpoints: one to receive requests, one to stream responses back over SSE. Multiple clients can connect to this same running process at once.

```python
# server.py
from flask import Flask, request, Response
import json, queue, threading

app = Flask(__name__)
sessions = {}  # session_id -> response queue, one per connected client

def handle(msg):
    if msg["method"] == "initialize":
        return {"jsonrpc": "2.0", "id": msg["id"],
                "result": {"capabilities": {"tools": {}}}}
    if msg["method"] == "tools/call" and msg["params"]["name"] == "add":
        a, b = msg["params"]["arguments"]["a"], msg["params"]["arguments"]["b"]
        return {"jsonrpc": "2.0", "id": msg["id"],
                "result": {"content": [{"type": "text", "text": str(a + b)}]}}

@app.route("/mcp/sse")
def sse_stream():
    # Auth check — this is the layer stdio never needed
    if request.headers.get("Authorization") != "Bearer sk-my-api-key":
        return "Unauthorized", 401

    session_id = request.args["session_id"]
    sessions[session_id] = queue.Queue()

    def event_stream():
        while True:
            response = sessions[session_id].get()   # blocks until server has something to push
            yield f"data: {json.dumps(response)}\n\n"

    return Response(event_stream(), mimetype="text/event-stream")

@app.route("/mcp/messages", methods=["POST"])
def receive_message():
    session_id = request.args["session_id"]
    msg = request.json
    response = handle(msg)
    if response:
        sessions[session_id].put(response)   # pushed to that client's SSE stream
    return "", 202

app.run(port=8080)
```

**The client** — opens an SSE connection to listen for replies, then sends requests as separate HTTP POSTs:

```python
# client.py
import requests, json, threading, sseclient  # pip install sseclient-py

BASE_URL = "https://mcp.example.com"
HEADERS = {"Authorization": "Bearer sk-my-api-key"}
session_id = "session-abc123"

# 1. Open the SSE stream in a background thread — this is how responses arrive
def listen():
    resp = requests.get(f"{BASE_URL}/mcp/sse?session_id={session_id}",
                         headers=HEADERS, stream=True)
    client = sseclient.SSEClient(resp)
    for event in client.events():
        print("Received:", json.loads(event.data))

threading.Thread(target=listen, daemon=True).start()

# 2. Send requests via POST — replies arrive asynchronously on the SSE stream above
def send(msg):
    requests.post(f"{BASE_URL}/mcp/messages?session_id={session_id}",
                   headers=HEADERS, json=msg)

send({"jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {}})
send({"jsonrpc": "2.0", "id": 2, "method": "tools/call",
      "params": {"name": "add", "arguments": {"a": 4, "b": 7}}})
```

#### What Actually Crosses the Network

```
CLIENT → SERVER  POST /mcp/messages?session_id=session-abc123
                 Authorization: Bearer sk-my-api-key
                 {"jsonrpc": "2.0", "id": 1, "method": "initialize", "params": {}}

SERVER → CLIENT  (on the open SSE stream)
                 data: {"jsonrpc": "2.0", "id": 1, "result": {"capabilities": {"tools": {}}}}

CLIENT → SERVER  POST /mcp/messages?session_id=session-abc123
                 Authorization: Bearer sk-my-api-key
                 {"jsonrpc": "2.0", "id": 2, "method": "tools/call", "params": {"name": "add", "arguments": {"a": 4, "b": 7}}}

SERVER → CLIENT  (on the open SSE stream)
                 data: {"jsonrpc": "2.0", "id": 2, "result": {"content": [{"type": "text", "text": "11"}]}}
```

#### Why This Example Makes the Bullet Points Concrete

- **"Long-running service, potentially serving many clients"** — `app.run()` starts one persistent process. The `sessions` dict is keyed by `session_id` specifically because this same server instance might have dozens of clients connected simultaneously, each needing its replies routed to the right SSE stream.
- **"HTTP POST carries client → server messages"** — every outgoing message is a distinct `POST /mcp/messages` call. Unlike stdio's single continuous pipe, each request here is its own independent HTTP call.
- **"SSE stream carries server → client"** — the client never polls for a response. It opens `/mcp/sse` once and keeps that connection open; the server pushes onto it (`sessions[session_id].put(...)`) whenever a reply, notification, or streamed chunk is ready.
- **"This is where auth becomes relevant"** — notice the `Authorization: Bearer ...` header appears on *every* request, and the server actively rejects connections without it (`401 Unauthorized`). Compare this to the stdio example last turn, which had no concept of authentication at all — there was no boundary to authenticate across.

Same caveat as before: a production MCP server would use the official SDK's HTTP+SSE transport implementation rather than hand-rolled Flask/session-queue plumbing — but this is the literal shape of what's happening underneath it: POST in, SSE out, session ID tying the two together, auth header gating the whole thing.

### Choosing Between Them

| | stdio | HTTP + SSE |
|---|---|---|
| **Server location** | Same machine, spawned as subprocess | Remote, reachable over network |
| **Setup complexity** | Minimal — just an executable path | Requires hosting, networking, often auth |
| **Auth needed?** | No | Often yes (API keys, OAuth) |
| **Multiple simultaneous clients?** | No — one subprocess per client | Yes — one server can serve many clients |
| **Typical use case** | Local dev tools, filesystem, local CLIs | Shared/hosted services, team tools |

Both transports carry the **exact same message format** underneath — switching transports doesn't change how tools, resources, or prompts are defined or negotiated. Only *how the bytes travel* changes.


## JSON-RPC 2.0 — The Message Format

Regardless of transport, every MCP message is a **JSON-RPC 2.0** object. JSON-RPC is a lightweight, transport-agnostic remote procedure call protocol — MCP didn't invent a new format, it adopted an existing, well-specified standard.

There are three message shapes to know:

### 1. Request — expects a response

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "get_weather",
    "arguments": { "city": "Tokyo" }
  }
}
```

- `id` — a unique identifier the sender uses to match the eventual response back to this request
- `method` — which operation is being invoked (MCP defines a fixed set of method names, e.g. `tools/list`, `tools/call`, `resources/read`)
- `params` — the arguments for that method

### 2. Response — the answer to a request

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [{ "type": "text", "text": "18°C, partly cloudy" }]
  }
}
```

- Same `id` as the originating request, so the caller knows which request this answers
- `result` on success — or, on failure, an `error` object instead:

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": { "code": -32602, "message": "Invalid params: city is required" }
}
```

### 3. Notification — fire-and-forget, no response expected

```json
{
  "jsonrpc": "2.0",
  "method": "notifications/tools/list_changed"
}
```

- **No `id` field** — this is the defining feature of a notification. Without an `id`, the receiver knows not to send a response at all.
- Used for one-way signals, like a server telling the client "my tool list just changed, you may want to refresh."

### Why JSON-RPC Specifically?

- **Simple and human-readable** — easy to log, debug, and inspect by hand
- **Transport-agnostic** — the same message shape works whether it's traveling over stdio or HTTP/SSE, which is exactly why MCP can support both transports without redefining the protocol twice
- **Well-established** — it's a pre-existing standard with mature tooling and clear semantics, rather than a bespoke format MCP would have to fully specify and defend from scratch

| **Example** | **Valid/Invalid** | **Reason** |
| --- | --- | --- |
| **[Your example](ca://s?q=Is_this_JSON_RPC_valid)** → ``{"jsonrpc":"2.0","id":2,"method":"tools/call","params":{"name":"add","arguments":{"a":4,"b":7}}}`` | ✅ Valid | Contains required fields (``jsonrpc``, ``id``, ``method``); ``params`` is allowed as object. Works if server defines ``tools/call``. |
| ``{"jsonrpc":"2.0","method":"subtract","params":[42,23],"id":1}`` | ✅ Valid | Standard request with array params. |
| ``{"jsonrpc":"2.0","result":19,"id":1}`` | ✅ Valid | Proper response object with ``result``. |
| ``{"jsonrpc":"2.0","error":{"code":-32601,"message":"Method ``not ``found"},"id":1}`` | ✅ Valid | Proper error response with standardized code. |
| ``{"method":"subtract","params":[42,23],"id":1}`` | ❌ Invalid | Missing ``"jsonrpc":"2.0"`` field. |
| ``{"jsonrpc":"2.0","method":42,"id":1}`` | ❌ Invalid | ``method`` must be a string, not a number. |
| ``{"jsonrpc":"2.0","result":19}`` | ❌ Invalid | Missing ``id`` field in response. |
| ``{"jsonrpc":"2.0","id":null,"method":"subtract","params":[42,23]}`` | ❌ Invalid | ``id:null`` is only allowed in response objects, not requests. |

## The Three Primitives a Server Can Expose

Once connected, a server can offer the client up to three kinds of capabilities. Understanding *which primitive to use for what* is one of the most important design decisions when building a server (covered in depth in Module 3).

### Tools — actions the model can call

Functions with a defined input schema that the model can invoke to **do something** or **compute something** — the model decides when to call them based on the conversation.

```json
{
  "name": "send_email",
  "description": "Send an email to a specified recipient",
  "inputSchema": {
    "type": "object",
    "properties": {
      "to": { "type": "string" },
      "subject": { "type": "string" },
      "body": { "type": "string" }
    },
    "required": ["to", "subject", "body"]
  }
}
```

Think of tools as **verbs** — `send_email`, `create_issue`, `run_query`. The model chooses to call them mid-conversation, and the server executes real logic (which may have side effects) and returns a result.

### Resources — data the model can read

Static or semi-static content exposed via a **URI**, meant to be read rather than executed. Resources are more like **files or records** than actions.

```json
{
  "uri": "file:///project/README.md",
  "name": "Project README",
  "mimeType": "text/markdown"
}
```

Examples: a specific file's contents, a database record, a cached API response. Resources are typically **attached to context** (either by the user explicitly picking one, or the host application deciding to include it) rather than invoked by the model choosing to call a function.

### Prompts — reusable prompt templates

Pre-written prompt structures the server provides, often parameterized, that a user or host can invoke to kick off a particular kind of interaction.

```json
{
  "name": "code_review",
  "description": "Review a code diff for bugs and style issues",
  "arguments": [
    { "name": "diff", "description": "The code diff to review", "required": true }
  ]
}
```

Think of prompts as **surfaced shortcuts** — a server exposing "here's a good way to ask me to do X" as a first-class, discoverable object, rather than the user having to know the right phrasing themselves.

### Quick Comparison

| Primitive | Analogy | Who decides to use it | Has side effects? |
|---|---|---|---|
| **Tools** | Verbs / actions | The model, during reasoning | Often yes |
| **Resources** | Files / records | The user or host application | No — read-only |
| **Prompts** | Templates / shortcuts | The user, explicitly selecting one | No |


## Capability Negotiation

Not every client supports every feature, and not every server exposes every primitive. Before any real work happens, client and server run a brief **handshake** to establish what's actually available in this specific connection.

```
Client                                          Server
  │                                                │
  │──── initialize (client capabilities) ────────▶│
  │                                                │
  │◀─── initialize response (server capabilities) ─│
  │                                                │
  │──── initialized (notification) ───────────────▶│
  │                                                │
  │              ... normal operation ...          │
```

**Step 1 — `initialize` request (client → server):** the client states which parts of the protocol it supports — e.g., whether it can render certain content types, whether it supports receiving sampling requests (see Module 7).

**Step 2 — `initialize` response (server → client):** the server replies with its own declared capabilities — which of the three primitives it offers (tools, resources, prompts), and any optional features it supports (e.g., whether its tool list can change dynamically at runtime).

**Step 3 — `initialized` notification (client → server):** the client confirms the handshake is complete and the connection is ready for normal operation. Being a notification, it expects no reply.

### Why Negotiate at All?

- **Servers evolve independently of clients.** A server might support a feature a particular client doesn't understand yet — negotiation lets both sides find the overlap instead of assuming full compatibility.
- **Avoids wasted calls.** If a server didn't declare `resources` capability, the client knows not to bother calling `resources/list` against it.
- **Enables graceful degradation.** A client can still use a server that only supports a subset of features — it simply won't offer the UI/behavior for capabilities that weren't declared.

This negotiation step is also *why* the Host → Client → Server model from Module 1 keeps each client-server pair isolated: capabilities are negotiated **per connection**, so Server A and Server B in the same host can expose completely different feature sets without conflicting.


## Putting It All Together

A full round trip, from cold start to tool call, looks like this:

```
1. Client launches/connects to Server         (via stdio or HTTP+SSE)
2. Client → Server: initialize                (JSON-RPC request)
3. Server → Client: capabilities response     (JSON-RPC response)
4. Client → Server: initialized                (JSON-RPC notification)
5. Client → Server: tools/list                 (JSON-RPC request)
6. Server → Client: [list of available tools]  (JSON-RPC response)
7. Model decides to call a tool
8. Client → Server: tools/call                 (JSON-RPC request)
9. Server → Client: tool result                (JSON-RPC response)
```

Every arrow in that sequence is a JSON-RPC message; every message travels over whichever transport (stdio or HTTP+SSE) the connection was established with.


## Key Terms Introduced in This Module

| Term | Definition |
|---|---|
| **Transport** | The underlying communication channel — stdio (local subprocess) or HTTP+SSE (remote network) |
| **JSON-RPC 2.0** | The message format every MCP exchange uses, regardless of transport |
| **Request / Response / Notification** | The three JSON-RPC message shapes — notifications are the only one with no `id` and no expected reply |
| **Tools** | Server-exposed functions the model can call to take action |
| **Resources** | Server-exposed data the model or host can read |
| **Prompts** | Server-exposed reusable prompt templates |
| **Capability negotiation** | The `initialize` handshake where client and server declare what they support before normal operation begins |
