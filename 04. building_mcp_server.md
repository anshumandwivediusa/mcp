# Module 3: Building an MCP Server

## 1. Choosing an SDK

Anthropic maintains official SDKs for building MCP servers in several languages. The two most mature are:

| SDK | Language | Notes |
|---|---|---|
| **Python SDK** | Python | Uses decorators (`@mcp.tool()`, `@mcp.resource()`) via the `FastMCP` high-level API; good for quick prototyping and data/ML-adjacent tools. |
| **TypeScript SDK** | TypeScript/Node.js | Explicit `server.tool()` / `server.resource()` registration; natural fit if your backend or existing tooling is already in Node. |
| **Others** | Kotlin, Java, C#, Swift, Go, Rust (community/official, varying maturity) | Useful if you're embedding a server inside an existing app written in that language. |

**Picking one** usually comes down to: what language is your backend already in, and do you want the fastest path (Python's `FastMCP` decorators) or the most explicit control (TypeScript's lower-level primitives).


## 2. Defining a tool

A tool has four core parts:

- **`name`** — a short, unique, machine-readable identifier (e.g. `get_weather`). This is what appears in `tools/list` and what the model calls in `tools/call`.
- **`description`** — natural-language text the model reads to decide *when* and *how* to use the tool. This is effectively your prompt to the model — vague descriptions cause misuse or non-use.
- **`input schema`** — a JSON Schema object describing the parameters the tool accepts (types, required fields, enums, descriptions per field). The model uses this to construct valid arguments.
- **`handler function`** — the actual code that runs when the tool is called. Receives validated arguments, performs the action (call an API, query a DB, run a calculation), and returns a result.

**Example (conceptual):**
```json
{
  "name": "get_weather",
  "description": "Get current weather conditions for a city",
  "inputSchema": {
    "type": "object",
    "properties": {
      "location": { "type": "string", "description": "City name" }
    },
    "required": ["location"]
  }
}
```
The handler is the function bound to this definition in your SDK of choice — decorated in Python, registered explicitly in TypeScript.

**Good practice:** keep tools narrow and composable rather than one giant multi-purpose tool; write descriptions as if explaining the tool to a new teammate, not just labeling it.


## 3. Defining a resource

Resources are read-only context, addressed differently from tools:

- **URI scheme** — resources are identified by a URI, e.g. `file:///path/to/doc.md`, `postgres://db/table/row`, or a custom scheme like `github://repo/issue/42`. The scheme signals what kind of resource it is and how the host might display it.
- **Content type (MIME type)** — declares what the returned content is (`text/plain`, `application/json`, `image/png`, etc.), so the host knows how to render or pass it to the model.
- **Read handler** — the function that, given a URI, fetches and returns the actual content. Unlike tools, resources aren't "called" with arguments the model makes up — they're typically listed (`resources/list`) and then read (`resources/read`) by URI, often attached by the host/user rather than chosen by the model.

**Example shape:**
```json
{
  "uri": "file:///reports/q3-summary.md",
  "name": "Q3 Summary",
  "mimeType": "text/markdown"
}
```


## 4. Local dev loop

A typical build-test cycle:

1. **Run the server over stdio** — for local development, stdio is the simplest transport: your server is just a subprocess that reads JSON-RPC from stdin and writes to stdout (logs go to stderr, never stdout, or you'll corrupt the protocol stream).
2. **Test with the MCP Inspector** — Anthropic's Inspector is a standalone dev tool (run via `npx @modelcontextprotocol/inspector <your-server-command>`) that gives you a UI to:
   - See your server's declared tools/resources/prompts
   - Manually invoke tools with test arguments
   - Inspect raw JSON-RPC requests/responses
   - Catch schema or handshake issues before wiring up a real host
3. Once it behaves correctly in the Inspector, connect it to a real host (e.g. Claude Desktop) via its config file, pointing to your local stdio command.

This loop — edit code → restart server → test in Inspector — is the fastest way to iterate before dealing with a full host integration.


## 5. Error handling and structured results

MCP distinguishes two very different kinds of failure:

- **Protocol-level errors** — malformed requests, unknown methods, invalid params. These use standard JSON-RPC error codes (`-32602`, `-32601`, etc.) and mean the *call itself* was invalid.
- **Tool execution errors** — the call was valid, but the underlying action failed (API down, rate limit, bad input value at runtime). These are **not** protocol errors — they're returned as a normal successful result with `isError: true`, so the model can see the error message as content and potentially retry, adjust arguments, or report back to the user.

```json
{
  "result": {
    "content": [{ "type": "text", "text": "Rate limit exceeded, try again in 30s" }],
    "isError": true
  }
}
```

**Why this matters:** if you throw a protocol-level error for a business-logic failure, the model never sees *why* it failed — it just sees a broken call. Returning a structured, readable error as content lets the model reason about it and adapt, which is usually what you want.
