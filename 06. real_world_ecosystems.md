# Module 6: Real-World Ecosystem

## 1. Reference servers

Anthropic and the MCP steering group maintain a small set of **reference servers** — not production tools, but worked examples showing SDK usage and protocol features:

| Server | Demonstrates |
|---|---|
| **Everything** | A kitchen-sink test server exercising prompts, resources, and tools together |
| **Fetch** | Web content fetching/conversion — a minimal single-purpose tool |
| **Filesystem** | Secure file operations with configurable access, plus dynamic **Roots** (client tells server which directories are in scope, at runtime) |
| **Git** | Read/search/manipulate a local repo — a good "wraps a CLI" example |
| **Memory** | Knowledge-graph-based persistent state across a session |
| **Sequential Thinking** | Structured, reflective multi-step reasoning as a tool |
| **Time** | Timezone conversion — about as simple as a tool gets |

These live in `modelcontextprotocol/servers` and are explicitly framed as *educational*, not hardened for production — you're meant to read the source, not just run the binary. A second tier — **official integrations** (GitHub, GitLab, Slack, Google Drive, Postgres, Redis, Sentry, etc.) — is maintained by the companies behind those platforms rather than the MCP steering group, and a third tier is the long tail of **community servers** (thousands of them, cataloged in places like the `awesome-mcp-servers` lists) of very mixed quality and trustworthiness.

## 2. The MCP Registry — how discovery works

The **official MCP Registry** (`registry.modelcontextprotocol.io`) launched to solve a specific problem: before it existed, there was no canonical way to find "what MCP servers exist" — just scattered GitHub READMEs and blog posts.

Its design is worth understanding because it's deliberately *not* an app store:

- **It's a metaregistry, not a package host.** The Registry stores metadata — pointers — not code or binaries. Actual packages still live on npm, PyPI, NuGet, Docker Hub, or GitHub/GitLab releases:
  ```
  MCP Registry: "weather-server v1.2.0 is at npm:weather-mcp"
  npm Registry:  [the actual code]
  ```
- **Every entry is a `server.json`** with a namespaced identity (`io.github.username/server-name` or `com.yourcompany/server-name`), install info (which package registry, version, args/env), and descriptive metadata.
- **Namespace authentication prevents impersonation.** Publishing under `io.github.username/*` requires GitHub OAuth; publishing under `com.yourcompany/*` requires proving domain ownership (DNS TXT record or HTTP challenge). You can't publish a server claiming to be `com.google/*` without actually controlling google.com.
- **It's not meant for end-users directly.** The Registry is built for **downstream aggregators** — subregistries and marketplaces (Anthropic's Connectors Directory, GitHub, Docker Hub, PulseMCP, Smithery, etc.) — to pull from on a regular cadence and re-present, often with their own curation, verification badges, or ratings layered on top. An individual client isn't expected to query the raw Registry API directly.
- **It's still a preview.** Anthropic notes it may see breaking changes or data resets before general availability — treat it as an emerging standard, not a finished one.

```
curl "https://registry.modelcontextprotocol.io/v0/servers?search=filesystem"
```

Notably, MCP itself is no longer solely an Anthropic project — the protocol has been contributed to open, multi-vendor stewardship (alongside OpenAI, Google, Microsoft, AWS and others as co-maintaining participants), which is part of why a neutral, vendor-agnostic registry mattered enough to build.

## 3. How this differs from — and complements — Anthropic's "Connectors"

This is a layering question, not a competition between two protocols — **Connectors are built on MCP**, not an alternative to it.

| | MCP (the protocol/servers) | Claude Connectors (the product) |
|---|---|---|
| **What it is** | An open specification + the servers implementing it | Anthropic's curated, one-click way to attach MCP servers inside Claude products |
| **Who maintains the catalog** | The community-run MCP Registry (vendor-neutral, metadata only) | Anthropic's Connectors Directory (verified + community listings, surfaced in-app) |
| **Setup** | Edit `claude_desktop_config.json` / `.mcp.json`, run a subprocess or point at a URL | Click "+" in chat → Connectors → authenticate via OAuth — no config file |
| **Where it runs** | Anywhere — your laptop (stdio) or any host you deploy to | Anthropic's cloud reaches out to a remote MCP server over HTTPS; custom connectors must be internet-reachable from Anthropic's infrastructure, not just your local network |
| **Trust model** | You vet servers yourself before configuring them | Anthropic-verified connectors go through a review; custom connectors (any MCP URL) are explicitly *not* vetted by Anthropic — same buyer-beware as self-configuring a raw server |
| **Examples** | `mcp-server-git`, `mcp-server-filesystem`, any server you write | Google Drive, Gmail, Google Calendar, GitHub, Slack, Microsoft 365 ship as ready-to-use first-party connectors with no setup beyond login |

So concretely: when someone says "Claude's Slack connector," that *is* an MCP server — Slack's, wrapped and verified by Anthropic, listed in the Connectors Directory, and presented with a one-click OAuth flow instead of a JSON config edit. The protocol underneath is identical to the `mcp-server-slack` you could configure by hand in Claude Desktop; Connectors is the packaging, discovery, and trust layer on top, purpose-built for non-developers. For local-only integrations (no public URL), Anthropic also supports **Desktop Extensions (`.mcpb`)** — an MCP server plus its dependencies bundled for one-click local install, sitting outside the remote-Connectors model entirely.
