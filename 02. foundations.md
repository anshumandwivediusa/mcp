# Module 1: Foundations — Why MCP Exists

## Overview

Before learning *how* to build with the **Model Context Protocol (MCP)**, it's worth understanding *why* it exists at all. This module covers the integration problem MCP was designed to solve, how it compares to prior approaches like function calling, and the basic architecture that every later module builds on top of.

**_Note Common protocol that defines how models talk to tools._**

## The Problem: N×M Integration Explosion

Before a standardized protocol, connecting an LLM application to external tools and data sources meant writing **custom, one-off integration code** for every single pairing.

Imagine 3 different LLM apps, each needing to connect to 3 different tools:

```
Apps:   [App A]  [App B]  [App C]
Tools:  [Slack]  [GitHub]  [Postgres]
```

Without a shared standard, this requires **9 separate integrations** — every app has to write its own Slack connector, its own GitHub connector, its own Postgres connector, and so on:

```
App A ↔ Slack     App A ↔ GitHub     App A ↔ Postgres
App B ↔ Slack     App B ↔ GitHub     App B ↔ Postgres
App C ↔ Slack     App C ↔ GitHub     App C ↔ Postgres
```

This is the **N×M problem**: `N` applications × `M` tools = `N×M` integrations, each maintained separately, each breaking independently, each duplicating logic that's fundamentally the same task — "let a model call a function and get structured data back."

## MCP's Pitch: Turning N×M into N+M

MCP proposes a **standardized protocol layer** that sits between applications and tools. If both sides agree to speak MCP:

- Tool builders write **one MCP server** — and it works with *any* MCP-compatible app
- App builders support **one MCP client** — and it can connect to *any* MCP-compatible server

```
Apps:   [App A]  [App B]  [App C]
                    │
              (MCP Protocol)
                    │
Tools:  [Slack]  [GitHub]  [Postgres]
```

This turns the relationship from **N×M** (every pair custom-built) into **N+M** (each side builds against the shared standard once). This is the same conceptual shift that USB brought to hardware peripherals, or that HTTP brought to client-server communication — a shared protocol eliminates the need for custom point-to-point wiring.

## How This Differs From "Function Calling" / Tool Use APIs

It's easy to confuse MCP with function calling — they're related, but operate at different layers:

| | Function Calling / Tool Use | MCP |
|---|---|---|
| **What it is** | An API feature — the model outputs a structured call to a function you defined | A **protocol** — a standardized way for *any* client and *any* server to discover and exchange tools/data |
| **Where the logic lives** | Inside your application code, tightly coupled to that one app | Inside a separate, reusable **MCP server** that any compliant client can connect to |
| **Reusability** | Tool defined for App A must be redefined if App B wants the same capability | Tool defined once in an MCP server is available to *every* MCP-compatible app |
| **Scope** | A single request-level capability (the model can call a function) | A full session-level connection (tools, resources, and prompts, negotiated up front) |

**Conceptually**: function calling is the *verb* (a model can invoke a function). MCP is the *standardized wiring* that determines how that function got discovered, described, and connected in the first place — MCP servers ultimately expose tools that get called via that same function-calling mechanism, but the discovery and connection layer is what's new.

## The Client-Server Architecture

MCP defines three roles that work together in every interaction:

```
┌─────────────┐        ┌─────────────┐        ┌─────────────┐
│    HOST     │        │   CLIENT    │        │   SERVER    │
│ (the app)   │───────▶│ (1:1 conn.) │───────▶│ (exposes    │
│ e.g. Claude │        │             │        │  tools/data)│
│  Desktop    │        │             │        │             │
└─────────────┘        └─────────────┘        └─────────────┘
```

- **Host** — the user-facing application (e.g., Claude Desktop, an IDE, a custom app). The host is what the person actually interacts with.
- **Client** — lives inside the host and maintains a dedicated **1:1 connection** to a single server. If a host connects to 3 servers, it runs 3 separate clients internally.
- **Server** — a lightweight program that exposes a specific set of capabilities: tools to call, data to read, or prompt templates to use. Servers are typically small, focused, and independent (a "Slack server," a "filesystem server," a "database server").

**Why 1:1 client-to-server, rather than one client juggling many servers?** This isolation keeps each connection's state, capabilities, and failures independent — if one server crashes or misbehaves, it doesn't take down the host's connection to every other server.

## Key Terms Introduced in This Module

| Term | Definition |
|---|---|
| **Host** | The application the end user interacts with (e.g., a chat app) |
| **Client** | The component inside the host that manages a single connection to one server |
| **Server** | A program exposing tools, resources, or prompts over the MCP protocol |
| **N×M problem** | The integration explosion that occurs without a shared standard |
| **Tools** | Functions/actions a server exposes for the model to call *(covered in depth in Module 2)* |
| **Resources** | Data a server exposes for the model to read *(covered in depth in Module 2)* |
| **Prompts** | Reusable prompt templates a server can provide *(covered in depth in Module 2)* |
