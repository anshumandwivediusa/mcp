# Module 7: Advanced Topics

## 1. Multi-server orchestration

A single host commonly holds several simultaneous MCP sessions — recall from Module 1: **one client instance per server, always 1:1**. Nothing in the protocol handles cross-server coordination for you; that logic lives entirely in the host.

```
Host application
 ├── MCP client A ↔ filesystem server
 ├── MCP client B ↔ github server
 └── MCP client C ↔ postgres server
```

Practical implications:

- **Tool namespace collisions.** If two servers both expose a tool called `search`, the host needs a disambiguation strategy — commonly prefixing by server name (`filesystem__search`, `github__search`) before presenting the combined list to the model.
- **Aggregated discovery.** The host calls `tools/list` on every connected server and merges the results into one flat list for the model — the model doesn't "know" which server a tool came from unless the host's naming convention encodes it.
- **Independent failure domains.** Each session negotiates and fails independently — one server crashing or timing out shouldn't take down sessions with the others. Design host-side error handling per-session, not globally.
- **Sequencing across servers is the model's job, not the protocol's.** "Read a file (filesystem server), then create an issue about it (github server)" is just two ordinary tool calls the model makes in sequence — MCP itself has no concept of a cross-server transaction or rollback.
- **Scale creates its own selection problem** — the same tool-count-hurts-accuracy issue from Module 5 gets worse fast: connect ten servers with ten tools each and the model is choosing from 100 options every turn. This is the practical reason connector pickers (Module 6) let you enable connectors *per conversation* rather than globally.

## 2. Sampling, in detail — inverting the usual direction

Every other flow in this doc goes **client → server** (the model wants something, so the client calls a tool). Sampling flips it: the **server** asks the **client's** model to think.

```
sequenceDiagram
  Client->>Server: tools/call
  Server->>Client: sampling/createMessage  ← server-initiated
  Client->>User: show prompt for approval
  User-->>Client: approve / edit / reject
  Client->>LLM: forward approved prompt
  LLM-->>Client: completion
  Client->>User: show response for approval
  User-->>Client: approve / edit / reject
  Client-->>Server: return approved result
  Server-->>Client: original tool result (using it)
```

Why this exists: it lets a server reason with an LLM — summarize a doc, decide next steps, draft a message — **without holding its own model API key**. The client stays the single point of control over cost, model choice, and safety.

```json
{
  "method": "sampling/createMessage",
  "params": {
    "messages": [{ "role": "user", "content": { "type": "text", "text": "Summarize this log file." } }],
    "systemPrompt": "You are a helpful developer assistant.",
    "modelPreferences": { "hints": [{ "name": "claude-sonnet" }], "intelligencePriority": 0.8 },
    "maxTokens": 300
  }
}
```

Key design points:

- **Two human checkpoints, by spec ("SHOULD").** Before the prompt goes to the model, the user can review/edit/reject it. After the model responds, the user can review/edit/reject the completion before it goes back to the server. The server never sees the raw model output unfiltered by the client.
- **The client picks the actual model.** `modelPreferences` is advisory (`hints`, plus `costPriority`/`speedPriority`/`intelligencePriority`) — the server suggests, the client decides based on what it actually has access to.
- **Not universally supported yet.** A client must declare the `sampling` capability during `initialize` for this to work at all — check for it before calling, and have a fallback path (e.g., ask the user directly, or skip the enhancement) if it's absent. Support across hosts is still uneven as the ecosystem matures, so don't build a server that hard-depends on sampling being available everywhere it runs.

## 3. Streaming responses over SSE

Two different things get called "SSE" in the MCP world — worth separating:

- **The legacy HTTP+SSE transport** (pre-2025-03-26): two endpoints, one for the client to POST to, one long-lived `GET` stream for the server to push over. This has been **superseded by Streamable HTTP**, which folds both directions into a single endpoint and lets a POST response *optionally* upgrade to an SSE stream.
- **SSE-as-a-streaming-mechanism inside Streamable HTTP**: when a client POSTs a request that will take a while or produce incremental output, the server can respond with `Content-Type: text/event-stream` and push a sequence of JSON-RPC messages (partial results, progress notifications, and finally the response) over that single connection, rather than blocking until everything is ready.

What this means practically:

- **Progress notifications** (`notifications/progress`) are the main thing riding on this stream — a long-running tool call can emit periodic progress updates before its final result, so a host can show a progress bar instead of a frozen spinner.
- **Prefer Streamable HTTP over the old dual-endpoint SSE transport in new work** — it's simpler to implement, works better behind load balancers (no long-lived GET to keep pinned to one backend instance), and is where new spec investment is going.
- **stdio doesn't need any of this.** Over stdio, "streaming" is just writing multiple newline-delimited JSON-RPC messages to stdout as they're ready — there's no separate SSE mechanism because there's no HTTP layer to begin with.

## 4. Versioning and backward compatibility

MCP versions the *specification itself* with a **date-stamped string**, `YYYY-MM-DD` — not semantic versioning. The date marks the last time a **backwards-incompatible** change was made; the version does **not** bump for backwards-compatible additions, which is deliberate: it means point-in-time interoperability is unambiguous without forcing a new version number on every minor feature add.

| Status | Meaning |
|---|---|
| **Draft** | In-progress, not ready for consumption |
| **Current** | Ready for use, may still receive backwards-compatible additions |
| **Final** | Complete and frozen — will never change again |

**Negotiation happens once, at `initialize`:**
```json
// Client sends the version it wants (normally its latest supported):
{ "method": "initialize", "params": { "protocolVersion": "2025-11-25", ... } }

// Server replies with the version it will use for this session —
// its own latest if it supports the client's version, otherwise its best match:
{ "result": { "protocolVersion": "2025-11-25", ... } }
```
Client and server **MAY** each support several versions internally, but **MUST** converge on exactly one for the session — if no shared version exists, the connection fails at the handshake, cleanly, rather than partially working.

Things worth knowing if you're maintaining a server long-term:

- **History has actually broken things.** JSON-RPC batching was added in `2025-03-26` and then *removed* again in `2025-06-18` — proof the spec will walk back a decision if it turns out to be a mistake. Don't assume "it's in the spec" means "it's permanent."
- **What's genuinely stable:** the JSON-RPC 2.0 wire format itself, and the core methods (`initialize`, `tools/call`, `resources/read`, `prompts/get`) — these have held across every revision so far.
- **Schema evolution, done safely:** add optional fields, never remove or repurpose required ones — a client holding a cached older schema should still be able to call your tool correctly.
  ```python
  # v1 clients still work — category is optional and new
  # search_products: { query: str, limit: int = 10 }
  # v2:
  # search_products: { query: str, limit: int = 10, category: str | None = None }
  ```
- **A newer spec is already in motion.** As of mid-2026, `2026-07-28` is a release candidate (not yet shipped — `2025-11-25` remains current). It's introducing a formal **Active / Deprecated / Removed** feature lifecycle with a minimum twelve-month deprecation window, and its draft already marks Sampling, Roots, and the old HTTP+SSE transport as heading toward deprecation — a preview of the deprecation policy above being applied to real features, not a reason to stop using them today.
